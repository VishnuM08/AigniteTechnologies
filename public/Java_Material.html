03 · JVM

JVM — Java Virtual Machine

The engine that actually runs your code

What it is

A virtual machine — a piece of software that pretends to be a computer. Java code doesn't run on your real CPU; it runs on the JVM, which runs on your CPU.

What it does

  • → Loads compiled .class files (bytecode)
  • → Verifies the bytecode is safe to run
  • → Translates bytecode to machine code (JIT)
  • → Manages memory — allocation and garbage collection
  • → Provides runtime services to your program

Key insight: JVM is platform-specific, but bytecode is not.

JRE — Java Runtime Environment

The complete package for running Java programs

What it is

The runtime needed to execute Java programs. If you only want to run Java apps (not write them), JRE is enough.

import java.util.Scanner;

public class TipCalculator {
    public static void main(String[] args) {

        // Read input

01 · Concept

What is control flow?

Without it, programs just go top to bottom — boring and rigid

By default, Java runs your code line by line, top to bottom. Control flow lets you:

  • 🔀 Branch — do different things based on conditions
  • 🔁 Loop — repeat actions multiple times
  • ⏩ Skip — jump over parts of code
  • 🛑 Stop — exit early when something happens
Think of it like a recipe with "if the cake is brown, take it out" — without conditions, you'd burn every cake.
A program with control flow
int age = 19;

if (age >= 18) {
    System.out.println("Can vote");
} else {
    System.out.println("Too young");
}

for (int i = 0; i < 3; i++) {
    System.out.println("Hi #" + i);
}

06 · Loop variants

while vs do-while

The difference: when is the condition checked?

while loop

Checks the condition before each iteration. If false on the first check, the body never runs.

while.java
int count = 5;
while (count > 0) {
    System.out.println(count);
    count--;
}
// prints: 5, 4, 3, 2, 1
Use when: you don't know in advance how many times to loop. E.g., reading until end of file.

do-while loop

Checks the condition after each iteration. Body runs at least once, no matter what.

doWhile.java
int n;
do {
    n = readUserInput();
    process(n);
} while (n != 0);

// asks at least once
Use when: the body must run at least once. E.g., menu prompts that ask before deciding to exit.

09 · Gotchas

Common pitfalls

Mistakes every Java beginner makes at least once

Infinite loop
forgot to update counter
int i = 0;
while (i < 10) {
    System.out.println(i);
    // forgot i++
}

Program never stops. Always make sure the condition can become false.

Missing break
switch fall-through
switch (n) {
  case 1: a();  // runs
  case 2: b();  // also runs!
}

Always add break (or use the new arrow syntax → in Java 14+).

Off-by-one
< vs <=
// only goes 0..9
for (int i=0; i<10; i++) {}

// goes 0..10 — 11 times!
for (int i=0; i<=10; i++) {}

Be deliberate about < vs <= — it's the #1 source of array bugs.

04 · Overloading

Method overloading

Same name — different parameter list — Java picks the right one

int add(int a, int b)         { return a + b; }
double add(double a, double b)   { return a + b; }
int add(int a, int b, int c)   { return a + b + c; }
String add(String a, String b)   { return a + b; }

// Java chooses based on the arguments you pass
add(1, 2);              // calls int version → 3
add(1.5, 2.5);          // calls double version → 4.0
add(1, 2, 3);           // calls 3-arg version → 6
add("Hi ", "Java");    // calls String version → "Hi Java"
✓ Allowed (different parameters)
• Different number of parameters
• Different types of parameters
• Different order of types
✗ Not allowed
• Only return type differs
• Only parameter names differ
• (Java needs to tell calls apart by their arguments)

08 · Real example

A complete class

Putting it all together

Account.java
public class Account {
    // === FIELDS (private — hidden) ===
    private String owner;
    private double balance;
    private static int nextId = 1;  // shared counter
    private final int id;            // can't change

    // === CONSTRUCTOR ===
    public Account(String owner, double initial) {
        this.owner = owner;
        this.balance = initial;
        this.id = nextId++;            // assign and increment
    }

    // === GETTERS — controlled read access ===
    public String getOwner()  { return owner;  }
    public double getBalance() { return balance; }
    public int    getId()     { return id;      }

    // === BEHAVIOR — controlled writes ===
    public void deposit(double amt) {
        if (amt > 0) balance += amt;
    }

    public boolean withdraw(double amt) {
        if (amt > 0 && amt <= balance) {
            balance -= amt;
            return true;
        }
        return false;
    }
}

07 · Abstraction

Abstract classes

Define what something IS, without saying exactly HOW

Abstract class — incomplete on purpose
public abstract class Shape {
    protected String name;

    // Regular method — works for all subclasses
    public void describe() {
        System.out.println("I am a " + name);

    // Abstract — no body. Subclasses MUST
    // provide their own implementation.
    public abstract double area();
}

// You can't do this:
// new Shape();   ← error, abstract class

public class Circle extends Shape {
    private double r;

        this.r = r;
        name = "Circle";

    public double area() {        // required

06 · When to use which

Picking the right tool

A rule of thumb for each

Use an interface when…

  • You describe a capability ("Comparable", "Runnable")
  • Unrelated classes need to share behavior
  • You want a class to implement multiple contracts
  • You're designing a plug-in or callback API
  • You don't need any shared state

Use an abstract class when…

  • You have a partial implementation to share (with fields and helper methods)
  • Subclasses are clearly related (Dog, Cat, Bird all are Animal)
  • You need a constructor or instance state
  • You want to use protected access
  • You'd write the same code in every subclass without one
In practice: use interfaces by default — they're more flexible. Reach for abstract classes when you genuinely need to share code AND state.

02 · try/catch

The try-catch-finally pattern

Three blocks, each with a clear job

Anatomy of exception handling
try {
    // 1. Code that might fail
    //    Run normally if no problem
    int n = Integer.parseInt(userInput);
    System.out.println(100 / n);

} catch (NumberFormatException e) {
    // 2a. Runs if user typed non-number
    System.out.println("Please enter a number");

} catch (ArithmeticException e) {
    // 2b. Runs if user typed 0
    System.out.println("Can't divide by zero");

} finally {
    // 3. ALWAYS runs — success or failure
    //    Use it for cleanup (close files, etc.)
    System.out.println("Done");
}

06 · Wildcards

Wildcards — ? extends and ? super

For flexible APIs that accept families of types

? extends — upper bound (read)
// Accepts a List of Number OR any subtype
void printAll(List<? extends Number> nums) {
    for (Number n : nums) {
        System.out.println(n);
    }
}

printAll(intList);     // ✓
printAll(doubleList);  // ✓

// You can READ as Number.
// You can NOT add new elements
// (Java doesn't know exact subtype).
? super — lower bound (write)
// Accepts a List of Integer OR any supertype
void addInts(List<? super Integer> list) {
    list.add(1);
    list.add(2);
    list.add(3);
}

addInts(intList);     // ✓
addInts(numberList);  // ✓
addInts(objectList);  // ✓

// You can WRITE Integers in.
// Reading gives back Object only.
PECS rule: Producer = Extends, Consumer = Super. If a list produces Ts (you read from it), use extends. If it consumes Ts (you write to it), use super.
07 · Type erasure

Type erasure — a quirk

What generics do (and don't do) at runtime

Java's generics work at compile time only. At runtime, all the type information is erased — a List<String> is just a List.

This is called type erasure. It exists for backwards compatibility with old code, but has some annoying consequences.

Practical impact: rare for app code. You'll mostly forget erasure exists until you try to do something exotic like new T().
Things you can't do because of erasure

    return new T();    // won't compile
}

// ✗ Can't check generic type at runtime

// ✓ Raw type works (but with warning)
if (obj instanceof List)

// ✗ Can't create generic arrays

// ✗ Can't have static field of type T
//   (each instance might pick different T)

04 · Buffering

Why buffering matters

Reading one byte at a time is painfully slow

Disk reads are thousands of times slower than memory reads. Without buffering, your program asks the OS for one character at a time — like sipping coffee through a needle.

Buffered classes read a big chunk at once (usually 8 KB), then serve characters from memory until they need to fetch the next chunk.

Rule: always wrap a raw FileReader in BufferedReader — or just use Files.newBufferedReader.
Wrapping for performance
// Slow — one char at a time from disk
FileReader raw = new FileReader("big.txt");
int c;
while ((c = raw.read()) != -1) {…}

// Fast — reads chunks into memory
BufferedReader br = new BufferedReader(
    new FileReader("big.txt")
);

// Best — modern API, already buffered
try (BufferedReader br =
        Files.newBufferedReader(Path.of("big.txt"))) {
    // read lines fast
}
05 · Real example

A complete file-processing example

import java.nio.file.*;
import java.util.*;

public class CsvTransform {
    public static void main(String[] args) throws Exception {
        Path input  = Path.of("people.csv");
        Path output = Path.of("adults.csv");

        try (var lines = Files.lines(input);
             BufferedWriter w = Files.newBufferedWriter(output)) {

            lines
                .skip(1)                                 // skip header
                .map(line -> line.split(","))
                .filter(cols -> Integer.parseInt(cols[1]) >= 18)
                .forEach(cols -> {
                    try {
                        w.write(String.join(",", cols));
                        w.newLine();
                    } catch (Exception e) {
                        throw new RuntimeException(e);
                    }
                });
        }

        System.out.println("Done — wrote " + output);
    }
}

//   • try-with-resources handles closing both streams

    int count = 0;

    void increment() {
        count++;   // not atomic!
    }
}

Counter c = new Counter();
for (int i = 0; i < 10; i++) {
    new Thread(() -> {
        for (int j = 0; j < 1000; j++) {
            c.increment();
        }
    }).start();
}
// Expected: 10,000
// Actual:   sometimes 9,847 or 9,923
// — increments get lost!
Why? count++ is THREE steps
// count++ really means:
int tmp = count;      // 1. read
tmp = tmp + 1;          // 2. add 1
count = tmp;          // 3. write

// If two threads interleave at step 1,
// they both read the same value, both add 1,
// and one of their writes overwrites the other.
//
// One increment is lost.
04 · Synchronized

Fixing it with synchronized

Make a method or block atomic

synchronized method
class Counter {
    private int count = 0;

    public synchronized void increment() {
        count++;
    }

    public synchronized int get() {
        return count;
    }
}

// Only ONE thread at a time can be inside
// any synchronized method on the same object.
//
// Other threads wait until it exits.
// Final count is always 10,000 — correct.
Better — use atomic classes
import java.util.concurrent.atomic.*;

class Counter {
    private AtomicInteger count = new AtomicInteger();

    public void increment() {
        count.incrementAndGet();
    }

    public int get() {
        return count.get();
    }
}

// Atomic classes use special hardware
// instructions — much faster than locks.
// Use them for simple counters/flags.
Note: synchronized protects against races but is slow if there's lots of contention. For complex coordination, use ReentrantLock, Semaphore, or concurrent collections like ConcurrentHashMap.
05 · Executors

ExecutorService — don't manage threads yourself

Thread pools handle reuse and scheduling for you

Modern way to run concurrent tasks

// 2. Submit work — fire-and-forget

    Thread.sleep(1000);
    return 42;
});

// 4. Shutdown when finished

//   newFixedThreadPool(N) — fixed N threads
//   newCachedThreadPool() — grows as needed, reuses idle

Virtual threads (Java 21+)

Millions of "threads" without breaking a sweat

Traditional Java threads map 1:1 to OS threads — expensive. A typical machine can only run a few thousand.

07 · Variable capture

Capturing variables from the enclosing scope

Lambdas remember the surroundings — with one rule

✓ Effectively final — works
public Runnable greeter(String name) {
    String msg = "Hello, " + name;

    return () -> System.out.println(msg);
}

// msg is captured into the lambda.
// The lambda "closes over" the variable
// (it's called a closure).
//
// Rule: captured variables must be
//   FINAL or EFFECTIVELY FINAL
//   (never reassigned after creation).
✗ Reassigned — won't compile
int counter = 0;

Runnable r = () -> counter++;    // ERROR!

counter = 5;
// Reassigning means counter is NOT
// effectively final.
//
// Why the rule? Multiple threads might
// run the lambda — without immutability,
// you'd have race conditions everywhere.
//
// Workaround: use AtomicInteger or
// wrap state in an object.
08 · Best practices

Lambda do's and don'ts

Keep them clean and readable

✓ Do

  • Keep lambdas short — 1 to 3 lines max
  • Extract to a named method if long or reused
  • Use method references when they read clearer
  • Use Java's built-in functional interfaces
  • Name parameters meaningfully (not just x, y)

✗ Don't

  • Cram 20 lines into one lambda
  • Throw checked exceptions inside lambdas
  • Mutate external state from a lambda (concurrency bugs)
  • Create custom interfaces when a built-in fits
  • Use lambdas just to look modern — clarity wins

01 · Concept

What is a stream?

A pipeline of operations on a sequence of values

A stream is not a data structure. It's a pipeline that flows data from a source through a series of operations to a final result.

Streams are lazy: nothing happens until a terminal operation triggers the flow. And streams are single-use: once consumed, you can't reuse them.

Think of it like SQL for collections: declare what you want, not how to compute it.
The same task — old vs new
// Old: imperative loop
List<String> result = new ArrayList<>();
for (String s : names) {
    if (s.length() > 3) {
        result.add(s.toUpperCase());
    }
}

// New: stream pipeline
List<String> result = names.stream()
    .filter(s -> s.length() > 3)
    .map(String::toUpperCase)
    .toList();

// Same result. The stream reads like
// English: "from these names, filter
// those longer than 3, uppercase them,
// collect into a list."

04 · The injection problem

Statement vs PreparedStatement

One of these is dangerous. Always use the other.

❌ Statement — vulnerable to SQL injection
String userInput = "'; DROP TABLE users; --";

Statement stmt = conn.createStatement();
String sql = "SELECT * FROM users WHERE name = '"
           + userInput + "'";
// SQL becomes:
//   SELECT * FROM users WHERE name = '';
//   DROP TABLE users; --'
//
// Your tables are gone.

ResultSet rs = stmt.executeQuery(sql);
✓ PreparedStatement — safe
String userInput = "'; DROP TABLE users; --";

PreparedStatement ps = conn.prepareStatement(
    "SELECT * FROM users WHERE name = ?");
ps.setString(1, userInput);

ResultSet rs = ps.executeQuery();
// The ? is a placeholder.
// The driver sends the query and the
// parameter SEPARATELY — the input
// is never interpreted as SQL.
//
// Safe regardless of input.
Rule: always use PreparedStatement with placeholders. Never concatenate user input into SQL strings. There are no exceptions.

06 · Transactions

Transactions — all or nothing

Group operations so they either ALL succeed or ALL roll back

Money transfer — classic example
try (Connection conn = DriverManager.getConnection(URL, USER, PASS)) {

    // Step 1: turn off autocommit so we control the boundary
    conn.setAutoCommit(false);

    try {
        // Debit one account
        try (PreparedStatement ps = conn.prepareStatement(
                "UPDATE accounts SET balance = balance - ? WHERE id = ?")) {
            ps.setBigDecimal(1, amount);
            ps.setInt(2, fromId);
            ps.executeUpdate();
        }

        // Credit the other
        try (PreparedStatement ps = conn.prepareStatement(
                "UPDATE accounts SET balance = balance + ? WHERE id = ?")) {
            ps.setBigDecimal(1, amount);
            ps.setInt(2, toId);
            ps.executeUpdate();
        }

        conn.commit();          // both updates are now permanent
    } catch (SQLException e) {
        conn.rollback();        // undo both — even partial work
        throw e;
    }
}

// Without a transaction, a crash between the two updates would
// debit one account without crediting the other. Money lost.
07 · Pools

Connection pools

Opening a connection is expensive — share them

Opening a JDBC connection involves TCP handshakes, authentication, and per-connection setup. It can take tens or hundreds of milliseconds.

A connection pool keeps a set of open connections ready to reuse. Your code "borrows" one, uses it, and returns it.

Standard choice: HikariCP — fast, reliable, the default in Spring Boot. Other options: Apache DBCP, c3p0.

08 · Real example

A real-world repository pattern

Wrap JDBC in a clean class

UserRepository.java
public class UserRepository {
    private final DataSource ds;

    public UserRepository(DataSource ds) { this.ds = ds; }

    public Optional<User> findById(int id) throws SQLException {
        try (Connection c = ds.getConnection();
             PreparedStatement ps = c.prepareStatement(
                 "SELECT id, name, age FROM users WHERE id = ?")) {
            ps.setInt(1, id);
            try (ResultSet rs = ps.executeQuery()) {
                if (rs.next()) {
                    return Optional.of(new User(
                        rs.getInt("id"),
                        rs.getString("name"),
                        rs.getInt("age")
                    ));
                }
                return Optional.empty();
            }
        }
    }

    public List<User> findAll() throws SQLException {
        List<User> users = new ArrayList<>();
        try (Connection c = ds.getConnection();
             PreparedStatement ps = c.prepareStatement("SELECT id, name, age FROM users");
             ResultSet rs = ps.executeQuery()) {
            while (rs.next()) {
                users.add(new User(rs.getInt(1), rs.getString(2), rs.getInt(3)));
            }
        }
        return users;
    }
}

02 · var

var — local-variable type inference

Less typing, same type safety (Java 10+)

Without var
Map<String, List<Integer>> scores
    = new HashMap<String, List<Integer>>();

ArrayList<Customer> customers
    = new ArrayList<Customer>();

        : scores.entrySet()) {

}
With var
var scores = new HashMap<String, List<Integer>>();

var customers = new ArrayList<Customer>();

for (var entry : scores.entrySet()) {
    // ...
}

// Still statically typed!
// Compiler infers the exact type
// from the right-hand side.

04 · Records

Records — immutable data classes (Java 14+)

One line replaces dozens of boilerplate

❌ Old: manually write everything
public final class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    public int x() { return x; }
    public int y() { return y; }

    @Override public boolean equals(Object o) {…}
    @Override public int hashCode() {…}
    @Override public String toString() {…}
}
// ~40 lines of code
✓ Record — same thing in one line
public record Point(int x, int y) {}

// You automatically get:
//   • final fields x and y
//   • constructor Point(int, int)
//   • accessor methods x() and y()
//   • equals(), hashCode(), toString()
//
// Records are immutable by default —
// no setters, fields are final.

// Use it
Point p = new Point(3, 4);
int x = p.x();
System.out.println(p);  // Point[x=3, y=4]
Use records for: DTOs, API responses, configuration, return values, anything that's "just data". You can still add methods if needed.
05 · switch expressions

Switch expressions (Java 14+)

Switch as a value-producing expression, with arrow syntax

❌ Old switch — statement only
String type;
switch (day) {
    case 1: case 2: case 3:
    case 4: case 5:
        type = "Weekday";
        break;
    case 6: case 7:
        type = "Weekend";
        break;
    default:
        throw new IllegalArgumentException();
}
✓ New switch — expression with arrows
String type = switch (day) {
    case 1, 2, 3, 4, 5 -> "Weekday";
    case 6, 7                -> "Weekend";
    default -> throw new IllegalArgumentException();
};

// Arrow syntax → no fall-through
// Multiple labels per case → no repetition
// Returns a value → assign directly
// Compiler checks exhaustiveness
Multi-line case with yield
int code = switch (status) {
    case "OK" -> 200;
    case "NOT_FOUND" -> 404;
    case "ERROR" -> {
        log("server error");
        yield 500;        // like "return" — gives the case its value
    }
    default -> -1;
};
06 · Pattern matching

Pattern matching

Test type AND cast in one step

❌ Old instanceof
if (obj instanceof String) {
    String s = (String) obj;
    System.out.println(s.length());
}

// Always: test → cast → use.
// Compiler can't tell they're related.
// The cast might fail if you mess up.
✓ Pattern instanceof (Java 16+)
if (obj instanceof String s) {
    System.out.println(s.length());
}

// Variable s is automatically:
//   • declared with the right type
//   • only in scope where the test
//     guarantees it.
Pattern switch (Java 21+) — by type
String describe(Object obj) {
    return switch (obj) {
        case Integer i when i < 0 -> "negative int";
        case Integer i              -> "int: " + i;
        case String s              -> "string of length " + s.length();
        case int[] arr             -> "int array of size " + arr.length;
        case null                  -> "null!";
        default                     -> "something else";
    };
}

// Replaces the old "chain of if-instanceof" idiom.
// More readable. Compiler checks exhaustiveness when types are sealed.
07 · Sealed classes

Sealed classes (Java 17+)

"Here is the COMPLETE list of subtypes I allow"

A sealed type hierarchy
// Only Circle, Square, and Triangle may extend Shape.
// Anyone else trying gets a compile error.
public sealed interface Shape
        permits Circle, Square, Triangle {}

public record Circle(double radius)       implements Shape {}
public record Square(double side)         implements Shape {}
public record Triangle(double b, double h) implements Shape {}

// Now the compiler KNOWS the full set.
// Combined with pattern switch, no default needed —
// it checks exhaustiveness for you:

double area(Shape s) {
    return switch (s) {
        case Circle c    -> Math.PI * c.radius() * c.radius();
        case Square sq   -> sq.side() * sq.side();
        case Triangle t  -> 0.5 * t.b() * t.h();
        // no default — compiler knows these are all cases
    };
}

// Add a new Shape type? Compiler errors at every switch
// that doesn't handle it. No more "we forgot to update X".
